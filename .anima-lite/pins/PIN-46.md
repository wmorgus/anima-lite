**"Movement map" renamed to "scaffold" 2026-07-14, same session** — read `movement.md` below as the pre-rename name for `scaffold.md`.

Spun out of PIN-45's scaffold discussion. Once `scaffold.md` (formal cause's deep tier, PIN-45) was placed as a spine file, the natural next question was whether material and efficient deserve the same top-level/deep-tier split — a data-lineage deep-tier under `material.md` (which entity/field is created/read/mutated at which point, and by what) and an invocation-graph deep-tier under `efficient.md` (what mechanism/trigger/handler literally fires at each step — distinct from formal's *logical* choreography of what-follows-what).

**Fork 1 — resolved.** Three separate deep files, each with its own coordinate scheme (safer, smaller, but risks the three grids drifting out of step with each other) vs. **one shared coordinate spine** with three column-families per cell. Operator's read: the second, explicitly — "it's like the actual spine under the spine." Single scaffold, no independent grids to fall out of sync, telos remains the top-level sentence governing the whole backbone.

**Fork 2 — schema, resolved.** One row per `(scenario, path, step)`, three columns (formal/material/efficient) — not a relational join across tables. Grounded in a worked example (anima-lite's own port pipeline, GATE-TELOS at step 4): formal cell = "ari-intake, GATE-TELOS check," material cell = "RESOLUTION.md; telos.md §1/§2, read-only," efficient cell = "inline main-agent check, synchronous halt." A join only earns its cost with real relational storage behind it (uno-blueprint has Supabase); anima-lite's substrate is markdown, no DB (`material.md` §2) — normalizing across tables solves a problem the storage medium doesn't have. Concretely: one markdown table per scenario, happy path as the default table, other paths get rows only where they diverge (gap = an empty column entry, not a missing row — uno-blueprint's own discipline). Extension proposed, not yet ratified: a `stub:0–3` field per scenario (reusing the feature-ledger's own vocabulary), so a scaffold can be founded with scenarios named but cells unfilled, deepening later — but see the skeleton/ledger distinction below for why this "deepening" is a refresh of an already-named coordinate, not new growth.

**The skeleton/ledger distinction — resolved, and this reframes how the backbone relates to existing machinery.** `telos.md`/`material.md`/`formal.md`/`efficient.md`/`scaffold.md` are all upfront, comprehensive-by-intent, exactly-one-per-repo, and — per `PHILOSOPHY.md`'s existing spine self-correction rule — refreshed, never accreted: "the spine stays a clean current-state snapshot... what changed and why is the record." The feature ledger (`features/<slug>.md`) is the one artifact in this family that's genuinely accretive: a new entry appears each time a feature lands, tied to a real event, never comprehensive by intent. So the five spine files (including the scaffold) are the **skeleton**; the ledger is the **flesh** that gets upholstered onto it over time. Without the scaffold, a pile of ledger entries has no guarantee of composing into a coherent whole-repo picture — nothing ever mapped the whole repo end to end (today's situation for ordinary port work-types). With the scaffold, a ledger entry that doesn't match what its coordinate called for becomes a checkable finding, same shape as GATE-HASH catching spine-vs-reality drift.

**Ledger-rooting field — resolved, concrete spec change identified.** Nothing in `ledger-spec.md`'s current fields (`Entry points`, `Full data flow`, `State machine`, etc.) points at a scaffold coordinate — so "a ledger entry can promote a scaffold cell's stub-depth" is currently unbuildable; there's no pointer to promote through. Fix: the ledger gains a field, `Scaffold coordinates: <scenario>/<path>/<step>[, ...]` — present when the repo has a scaffold, absent (not "none") when it doesn't, same graceful-degradation posture as the playwright block or `Schema deps:`. And it should run both directions, matching the bidirectional-audit instinct already in this harness (same shape as `lives-in:`): a filled scaffold cell carries `Filled by: features/<slug>.md`, so walking from either artifact finds the other. This is what makes the tier-1 reconsideration below actually computable instead of requiring a human to eyeball the whole scaffold.

**Telos-growth propagation through the backbone — resolved as a borrowed mechanism, not new machinery.** A telos shift on an arete statement is the existing drift/growth fork (ruling 4, inherited via ruling 2) — drift fixes the scaffold/code, growth is its own workstream, never bundled, explicit ratification, provoking change stays blocked until resolved. What was open: once growth is ratified, what happens to everything scaffolded off the old statement? Not automatic invalidation (too blunt, violates detection≠verdict) and not silent assumption of no-impact (exactly the unexamined drift this harness exists to catch). Resolution: borrow ripple's two-tier CONTRACT-BREAK reopen directly (`reorient/ripple.md` ruling 4) — **tier 1**, the moment a system-level arete-statement growth is ratified, every cascaded component statement *and* every scaffold scenario gets a mandatory reconsideration pass ("does this still follow from the new parent?"), judgment logged even when the answer is no-rework-needed; **tier 2**, only re-founding/re-walking where that judgment concludes the scenario/statement actually depended on what changed. Remaining gap: tier 1 needs to know *which* cells/statements derive from the changed clause to be computable rather than manual — that's a new edge kind on top of PIN-37/40's existing `lives-in:`/`relates-to:` tags (a scaffold cell or cascaded statement citing which parent-telos clause it derives from), not yet designed.

**Corrected 2026-07-14, later same session:** whether every repo's spine gets this backbone is resolved, not open — `ari-map/SKILL.md` already runs an unconditional "comprehensive feature map" step every probe (the walk that seeds the feature ledger), and that walk is most of what a scaffold needs; PIN-45 now states this as always-produced, depth-gated by work-type rather than existence-gated. This backbone (the three-column-family grid) inherits the same shape: always available, depth is what arete's rigor actually buys.

Not yet designed: how this interacts with `ari-map`'s existing per-cause file structure without breaking the `telos.md`-as-entry-point / cause-files-as-reference-depth pattern; whether this is a `HARNESS.md` §2 spec-ownership addition (a new canonical shape doc, likely owned by `ari-map/SKILL.md`'s Output section); the telos-derivation edge kind that makes tier-1 reconsideration computable (named above, unbuilt); the `ledger-spec.md` field change itself (named above, unbuilt).
