# Formal: lumilo-bridge (lumilo-bridge)
(Reference depth — see telos.md for entry point and commit hash)

## §1 Layered architecture
Backend (`graphql-server/`): resolvers (`src/resolvers/*.ts`) → services (`src/services/*`, business logic) → Mongoose models (`src/services/mongodb/models/*`) for persistence, plus Redis (`src/services/redis/*`) for cache/pub-sub/dedup. Frontend (`bridge-frontend/`): standalone Angular components under `src/app/features/*`, `apollo-angular` split-link (HTTP for queries/mutations, `graphql-ws` for subscriptions), codegen'd GraphQL types/services under `src/app/graphql/generated`. `graphql-server` also serves the built Angular SPA as static files from `graphql-server/public/teacher/` (ng build outputs there, base-href `/lumilo-bridge/teacher/`) — one deployable unit, two source trees. (code-derived)

## §2 Module boundaries
Ingestion sources (GraphQL mutation resolver, Kinesis consumer, TutorShop WS client) are kept separate from the shared ingestion/detector/persistence logic — each source is a thin adapter that produces a `LumiloEvent` and calls `commonIngestionService.ingest()`; this boundary is the load-bearing one for the upcoming Kinesis-direct ripple (see §3 Data flow, §4). Auth providers (`src/services/auth/providers/*`) are kept separate from the shared `authMiddleware`/`buildContext`, except REST routes (`src/routes.ts`), which reimplement authorization ad hoc rather than sharing that guard. (code-derived)

## §3 Dominant patterns

### Backend
Services are mostly singleton classes with a module-level `export default new X()` (e.g. `commonIngestionService`, `mathiaKinesisService`, `tutorshopWsService`, `redisService`, `mongoDBService`), while detectors are mostly plain exported functions — there is no stated rule for which shape applies where; this is a named inconsistency, not two intentional patterns. Detector dispatch from `commonIngestionService.ingest()` is fire-and-forget (`dispatchDetectors()` is not awaited) — errors in a detector never propagate back to the ingestion caller. Idle detection is the outlier: it runs on an independent 15-second poll loop (`startIdleDetectionSchedule`) rather than through per-event `dispatchDetectors()`, so "all detectors run on the normalized event stream" is not uniformly true.
`Seams: singleton-class-vs-plain-function has no documented rule (commonIngestionService/mathiaKinesisService/tutorshopWsService are classes; most detectors are functions); idle detection bypasses the per-event detector dispatch used by the other detectors.`

### Frontend
Standalone Angular components (no NgModules), each feature area under `src/app/features/<area>/`, Apollo Angular services generated via codegen from `.graphql` query files under `src/app/graphql/queries/`. Sampled files (apollo config, a query file, a feature component) show this pattern applied consistently.
`Seams: none found in sampled files.`

### Data flow
Three ingestion sources converge on one seam: normalize to `LumiloEvent` → `commonIngestionService.ingest(event)`. Confirmed live: `dataIngestion.ts` resolver (source `MATHIA_GRAPHQL`) — this is the current production path, arriving as GraphQL mutations (`addAttempt`, `addSkillChange`, `addAttemptsBatch`, `addSkillChangesBatch`, `addContextChange`, `addContentChange`, `addPositionChange`) defined in `deviceInteractionSchema.graphql`/`mathiaDataSchema.graphql` and implemented in `graphql-server/src/resolvers/dataIngestion.ts`. Confirmed built-but-gated: `mathiaKinesis.service.ts` (source `MATHIA_KINESIS`) — see §4. Confirmed scaffold/dormant: `tutorshopWs.service.ts` (source `TUTORSHOP`) — field names throughout are marked `// TODO: confirm with TutorShop team`, gated on `TUTORSHOP_WS_URL`; this is the weakest of the three legs, wired into startup but not production-ready. The three legs are not symmetric: the TutorShop path calls `runPlatformDetectors()` before `ingest()`, while the GraphQL and Kinesis paths call `ingest()` alone — an asymmetry worth resolving before the ripple, not just porting as-is.
`Seams: the three ingestion adapters share one target call (ingest()) but differ in liveness (GQL live / Kinesis built-but-gated / TutorShop stubbed) and in call shape (TutorShop invokes runPlatformDetectors() pre-ingest, the other two do not).`

## §4 Seam protocols

**GraphQL mutation → LumiloEvent (current production path).** `graphql-server/src/resolvers/dataIngestion.ts`: validates required fields, checks `LUMILO_CLASS_IDS` allow-list, dedupes via `redisService.checkAndRegisterMessage(messageId)`, builds a `LumiloEvent` with `source: 'MATHIA_GRAPHQL'` and `sourceEventId: messageId`, calls `commonIngestionService.ingest(event)`. `classId`/`studentId` arrive already as lumilo-internal IDs — no identity translation happens at this adapter. This is the boundary the upcoming ripple removes. (code-derived)

**Kinesis record → LumiloEvent (built, gated, not yet the primary path).** `graphql-server/src/services/mathiaKinesis.service.ts`: a persistent per-shard poll loop (`GetRecordsCommand`, `Limit: 100`, `KINESIS_POLL_INTERVAL_MS` default 1000ms) started unconditionally at server boot (`index.ts` line ~229) and no-op'ing only if `KINESIS_STREAM_NAME` is unset. Shard iterators are checkpointed to Redis (`SHARD_ITERATOR_KEY`) for restart recovery; on error, iterators are re-fetched after a 5s backoff. `decodeRecord()` parses the raw record body as JSON and requires `classId`, `studentId`, `eventType`, `payload` to already be present — i.e. it expects the Kinesis record to already be shaped like (most of) a `LumiloEvent`, stamping only `source` and `sourceEventId` (falling back to the Kinesis `SequenceNumber`) if missing. Deduping and `ingest()` call are identical in shape to the GraphQL path. **Named finding:** `getShardIds()` does not call `ListShards` — it synthesizes shard IDs from `DescribeStreamSummary`'s `OpenShardCount` using the standard `shardId-000000000000` naming pattern, with an explicit code comment flagging this as a simplification that will not survive shard splits/merges (`// TODO: replace with ListShards if shard splits/merges need to be handled at runtime`). This is a code-derived, self-acknowledged gap, not a doc-vs-code contradiction. **Named finding (doc-vs-code):** `common-ingestion-format.md` (root) lists "Define Kinesis shard/checkpoint strategy" as an open item — the code above shows a working, if simplified, checkpoint strategy already implemented. (code-derived)

**Whether the Kinesis record shape this consumer expects matches what the actual upstream producer emits is a cross-repo question this spine does not answer** — out of scope for a single-repo probe; flagged here only so ari-argue-rhetoric knows to check it before treating `mathiaKinesis.service.ts` as ripple-ready.

**LumiloEvent → Mongo write.** `commonIngestionService.ingest()` (`graphql-server/src/services/commonIngestion.service.ts`): resolves student/class identity cache-first (Redis, falling back to Mongo, creating the document if missing), then routes by `eventType` to `handleAttempt`/`handleSkillChange` (both use hand-rolled `updateOne` calls with `arrayFilters` against nested embedded arrays, not Mongoose's higher-level upsert helpers, plus a fallback create path if the array-filter update matches nothing); `NAVIGATION` events are currently a no-op stub. After the write, relevant Redis caches are invalidated and `dispatchDetectors()` is fired without being awaited. (code-derived)

**Detector result → Redis pub/sub → GraphQL subscription → client.** Status-changing detector output is published via `publishStatusEvent` (`src/services/redis/pubsub.helper.ts`) over `graphql-redis-subscriptions`, consumed by the `Subscription.listenOnClasses` resolver (`src/resolvers/client.ts`) with enrollment-based, fail-open filtering, delivered to subscribers over `graphql-ws`. Consumers are the Angular web client and AR/Unity devices connecting to the same WebSocket endpoint (`/lumilo-bridge/subscriptions`). (code-derived)

**Auth.** Five providers exist under `src/services/auth/providers/` (bridge-token, cltoken, google, jwt, lstoken) funneled through one shared `authMiddleware`/`buildContext` → `AuthService.buildAuthContext` for GraphQL and WebSocket connections. `TRIPLE_AUTH_INTEGRATION.md` documents 3 auth methods; code has 5 — a doc-vs-code gap, same category as the Kinesis-checkpoint doc gap above. (README-stated doc; code-derived count) REST routes (`src/routes.ts`) do not go through this shared middleware and instead reimplement authorization per-handler. `BYPASS_AUTH=true` disables auth in any environment including production (only a log warning fires) and the bypass check is duplicated independently in both the Express middleware and the WebSocket `buildContext` path rather than sharing one source of truth. (code-derived)

## §5 Named findings

- (README-stated vs. code-derived) `common-ingestion-format.md` is titled "Draft / In Progress" on branch `wip/prep-for-tutorshop` and lists the Kinesis checkpoint strategy as TBD; the current branch (`wip/common-ingestion-prep`) already has a working Redis-checkpointed Kinesis consumer. Treat the doc as historical design intent, not current-state truth. `lives-in: common-ingestion-format.md, graphql-server/src/services/mathiaKinesis.service.ts`
- (code-derived) `mathiaKinesisService.start()` and `tutorshopWsService.start()` are both called unconditionally in `index.ts`'s `startServer()`, each independently self-gating on its own env var; a future direct-Kinesis-only ingestion design should account for both already being wired into the same startup/shutdown lifecycle, not just one. `lives-in: graphql-server/index.ts`
- (code-derived) Detector dispatch from `ingest()` is fire-and-forget; a Kinesis-direct path that needs at-least-once processing guarantees for detector side effects (not just the Mongo write) does not get them from the current shared pipeline. `lives-in: graphql-server/src/services/commonIngestion.service.ts`
- (code-derived) The student-status enum (`IDLE|CONTENT_HELP|UNPRODUCTIVE_STRUGGLE|SYSTEM_MISUSE|WORKING|DOING_WELL|OFFLINE|LATE_START`) is duplicated verbatim across three files with no shared constant; a comment in `src/services/aggregation/types.ts` references a `'STUCK'` status that does not exist in any live enum — likely a stale/informal name, not a live status. `lives-in: graphql-server/src/services/mongodb/models/student.model.ts, graphql-server/src/services/mongodb/models/studentClassStatus.model.ts, graphql-server/src/types.ts`
- (code-derived) Two Mongo driver majors coexist in the repo: `graphql-server` pins `mongodb ^5.7.0`, the standalone `scripts/` toolkit pins `mongodb ^6.16.0` — separate dependency trees, not a shared workspace.
