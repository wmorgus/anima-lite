# Efficient: realtime-event-provider (realtime-event-provider)
(Reference depth — see telos.md for entry point and commit hash)

## §1 Build tooling
- Maven (rep-flink): `mvn clean package` produces `rep-flink/target/rep-flink.jar` (finalName `rep-flink`, per pom.xml `<build><finalName>`), a fat JAR via maven-shade-plugin 3.6.0, excluding SLF4J/log4j from the shaded artifact (provided/excluded at the default profile; the `local-execution` and `local-testing` Maven profiles override this to include them for local runs).
- npm (rep-delivery, rep-key-rotation, rep-simulator): each has its own `package.json` with a `build` script that runs `npm install --production` then zips the directory into a Lambda deployment package (`<name>.zip`), excluding tests/logs/`.git`/markdown. rep-key-rotation additionally has a `jest`-driven `test`/`test:coverage` script; rep-delivery and rep-simulator have no test script wired into `package.json` (rep-delivery has a `tests/` dir with its own separate `package.json` and `package-lock.json` — a second, isolated npm project for running rep-delivery's tests, not invoked by the parent's `build` script).
- Jenkins build agent requirement (`.jenkins/build.Jenkinsfile`): `nodejs22`, `jdk21`, `maven`, 4096 CPU / 8192 MB memory — this is a hidden constraint not mentioned in CLAUDE.md, which says "Java 11+" and "Node.js 14+" for local dev; the CI agent runs newer majors than the stated local minimums.

## §2 Key targets
- `cd rep-flink && mvn clean compile` / `mvn package` — compile / fat-jar build.
- `cd rep-flink && mvn test` (whole suite) or `mvn test -Dtest=<ClassName>` (single test) — unit tests; `local-testing` Maven profile activates `FLINK_LOCAL_TEST=true` + LocalStack/Redis system properties via surefire.
- `./local-test.sh --mode {unit|integration|docker|graphql|full}` — the repo's own composite local-verification entry point; `graphql` mode is the one CLAUDE.md calls out as testing the full simulator→Flink→delivery→GraphQL chain end-to-end.
- `node rep-simulator/test-data-generator.js` — CLI event generator for file/console/LocalStack-Kinesis output, independent of the Lambda simulator.
- `cd rep-delivery && REP_LOCAL_MODE=true REP_LOCAL_GRAPHQL_ENDPOINT=... node index.js --local-consumer` — runs rep-delivery as a standalone polling Kinesis consumer (not via Lambda event source mapping) for local debugging.

## §3 CI/CD
Two Jenkinsfiles, not one pipeline:
- `.jenkins/build.Jenkinsfile`: `multibranchBuildV2` job. Builds `rep-flink` via Maven (`mvn -B package -DskipTests -am -f <modules>/pom.xml` — tests are skipped in the CI build step itself; `mvn test` is a separate, manually-invoked target per CLAUDE.md) and `rep-delivery`/`rep-simulator`/`rep-key-rotation` via `npm ci --no-audit && npm run build`, each conditioned on whether that project's files changed, a user-requested build param, or first-build-on-default-branch. Successful builds are versioned and uploaded to S3 (`env.s3ArtifactBucket`, prefix `realtime-event-provider/<project>/<version>/`). On the default branch (or when `deploy-realtime-event-provider` param is set), triggers the `Realtime-Event-Provider-Deploy` job with `environment=qa` and each built project's version.
- `.jenkins/deploy.Jenkinsfile`: takes the built-artifact versions, checks out the external AWS infrastructure repo (`cloudFormation.checkoutAwsInfrastructure()`), and runs `cloudFormation.createStack(...)` per profile — `rep-delivery` (bundled with `rep-key-rotation` and `rep-flink` as extra version parameters `KeyRotationLambdaVersion`/`FlinkVersion` on the *same* stack) and `rep-simulator` (its own independent stack). This confirms: **rep-delivery, rep-key-rotation, and rep-flink deploy as one CloudFormation stack**; rep-simulator is deployed separately. Stack configs are loaded from `baseDir: "lambdas/realtime-event-provider"` in that external infra repo — this repo's own `infrastructure.yaml` is not what's read here (see formal.md §5).
- Slack notifications and a "cortex" release-tracking call (`serviceEntityId: "realtime-event-provider-lambda"`) fire on both build and deploy.
- Named finding: `CLAUDE.md`'s documented "Jenkins CI/CD Integration" section (its example `environment = env.BRANCH_NAME == 'main' ? 'prod' : ...` groovy snippet and multi-stage pipeline example) does not match either actual Jenkinsfile in this repo — the real files use `multibranchBuildV2`/`cloudFormation.createStack` helper functions from a shared Jenkins library, not the raw `pipeline { stages { ... } }` DSL shown in the doc. (README-stated pattern; code shows a materially different, higher-level pipeline abstraction.)

## §4 Branching model
Inferred only from Jenkins config, not independently confirmed: `env.BRANCH_NAME == env.defaultBranch` triggers automatic QA deploy on default-branch builds (build.Jenkinsfile); PR builds (`isPR()`) run checks only, no deploy. No branch-naming convention (e.g. `release/*`) was found in the actual Jenkinsfiles — CLAUDE.md's `develop`/`release/*`/`main` three-environment branching example is not reflected in the real `deploy.Jenkinsfile`, which deploys everything to a single `environment` job parameter (defaulting to `qa`) rather than branch-inferring dev/staging/prod. Named as a doc/code gap, not resolved.

## §5 Deploy path
Real path (code-derived): git push to default branch (or manual build param) → `build.Jenkinsfile` builds changed Maven/Node projects → artifacts versioned and uploaded to S3 → `Realtime-Event-Provider-Deploy` job triggered with `environment=qa` → `deploy.Jenkinsfile` checks out the external AWS-infrastructure repo → `cloudFormation.createStack` updates the `rep-delivery-qa` stack (bundling rep-delivery + rep-key-rotation + rep-flink versions) and the `rep-simulator-qa` stack.
Local/reference path (present in this repo but not confirmed as the real prod mechanism): `./deploy.sh --environment {dev|staging|prod}` → builds rep-flink via Maven, uploads the jar to a self-created S3 bucket, deploys `infrastructure.yaml` directly via `aws cloudformation create-stack/update-stack`, separately packages+updates the simulator Lambda. This script has internal inconsistencies (see formal.md §5) suggesting it has drifted from actual use.
