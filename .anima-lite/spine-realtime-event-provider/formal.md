# Formal: realtime-event-provider (realtime-event-provider)
(Reference depth — see telos.md for entry point and commit hash)

## §1 Layered architecture
Two independent pipelines sharing one repo:
1. **Ingest/enrich (rep-flink, rep-simulator)**: Kinesis source(s) → Flink filter/keyBy/broadcast-enrich → Kinesis sink. Java, long-running streaming job on AWS Kinesis Data Analytics.
2. **Deliver (rep-delivery, rep-key-rotation)**: Kinesis-triggered Lambda → per-message-type transform → JWT-signed GraphQL mutation to N downstream apps. Node.js, event-driven Lambda.
These two layers meet at exactly one seam: the output Kinesis stream (`rep-delivery-input-stream-{env}`). rep-key-rotation is a third, side-channel Lambda (Secrets Manager rotation trigger) that only rep-delivery depends on (for its signing key), not part of either pipeline's data path.

## §2 Module boundaries
- `rep-flink/` — owns Kinesis consumption from CTP10 + simulator, Redis session enrichment, routing-config broadcast, Kinesis production to the output stream. Package root `com.amazonaws.services.kinesisanalytics`.
- `rep-delivery/` — owns Kinesis consumption from the output stream, tenant/class routing, GraphQL mutation dispatch, JWT signing (calls into `auth/jwtService.js`, `services/secretsService.js`, `services/httpService.js`).
- `rep-key-rotation/` — owns RSA keypair generation/rotation for the JWT signing key rep-delivery uses; talks only to Secrets Manager.
- `rep-simulator/` — owns synthetic event generation onto the simulator input stream; talks only to Kinesis (PutRecord), no knowledge of enrichment or delivery.
- `config/` — shared environment JSON (routing config, consumed by both `deploy.sh` at deploy time and `rep-delivery/config/index.js` at Lambda cold-start) plus a JSON Schema that has drifted from the config files it's meant to validate (see §5).
- `infrastructure.yaml` / `deploy.sh` — declared as the deployment mechanism by CLAUDE.md, but the real prod deploy path is Jenkins + an external infrastructure repo (see efficient.md §3, §5). Treat these two files as local/reference-only unless proven otherwise for a given environment.

## §3 Dominant patterns

### Backend (rep-flink, Java)
Service-oriented within a single Flink job: `RepMain` wires sources → filter → keyBy → broadcast-connect → `StreamProcessorOrchestrator` (a `KeyedBroadcastProcessFunction`) → sink. The orchestrator delegates per-message-type work to a `MessageProcessorFactory`-selected `MessageTypeProcessor` (one processor per message type: `IdentityContextChangeProcessor`, `EnrichmentProcessor` reused across the other five types), backed by three injected services (`RedisSessionService`, `MetricsService`, `RoutingValidator`). Configuration is loaded once via `ConfigurationService` with a three-tier fallback (AWS runtime → mock runtime via reflection → hardcoded fallback constants) — this fallback chain is itself a defensive-handling signal: the authors feared the app failing to start more than they feared using slightly-wrong config.
`Seams: none found` — every message type flows through the same orchestrator→factory→processor chain; no exceptions observed in RepMain.java, StreamProcessorOrchestrator.java, or MessageProcessorFactory.java.

### Frontend
n/a — no UI in this repo.

### Data flow
`ctp10-{env}-pipeline` ∪ `rep-simulator-stream-{env}` (Kinesis, JSON records with `@class` type discriminator) → `EventMessageDeserializationSchema` (parses `@class`→messageType, `session_id`, and IdentityContextChangeMessage's camelCase `tenantId`/`groupId`/`userId`; retains full original JSON) → filter (`MessageTypeFilterFunction`, keeps only the 6 known types) → `keyBy(sessionId)` → connect with broadcast `RoutingConfig` stream → `StreamProcessorOrchestrator` (Redis-enrich + validate) → `EventMessageSerializationSchema` (emits snake_case `tenant_id`/`group_id`/`user_id`/`class` + passthrough original fields) → `rep-delivery-input-stream-{env}` (Kinesis) → **[current lambda-mediated leg, see §4]** → Lambda decodes base64 Kinesis record, JSON-parses, reads `record.tenant_id` + `record.group_id` (aliased as classId) → `getMatchingApplications` (OR-match against each configured app's `validTenantIds`/`validClassIds`) → per matching app, transform (Mathia-shape → GraphQL variables) → JWT-sign (RS256, audience = app name) → POST GraphQL mutation to that app's `endpoint`.
`Seams: the input-side camelCase (tenantId) vs output-side snake_case (tenant_id) field naming is an intentional transform at the enrichment boundary, not an accidental inconsistency — confirmed by reading both schema classes; named here so a port doesn't "fix" it into uniform casing.`

## §4 Seam protocols
- **Flink → output Kinesis stream**: JSON via `EventMessageSerializationSchema` (code-derived, §3 Data flow above). This is the seam the upcoming ripple cares about — it is unchanged by removing the Lambda leg, since the Flink side has no knowledge of what consumes its output stream. A new direct consumer (lumilo-bridge's planned service) must parse this exact shape.
- **Output Kinesis stream → rep-delivery Lambda**: `AWS::Lambda::EventSourceMapping` (`infrastructure.yaml`, `BatchSize: 100`, `MaximumBatchingWindowInSeconds: 5`, `StartingPosition: LATEST`) — standard Kinesis-to-Lambda push, base64-encoded `record.kinesis.data` per record, batched. This is the leg the ripple removes. Its removal requires nothing on the Flink/output-stream side; it only removes rep-delivery's consumption of the stream and (per-app) its GraphQL dispatch to that app.
- **rep-delivery → downstream GraphQL apps (lid-backend, lumilo-bridge)**: HTTP POST, `{query, variables}` body, `Authorization: Bearer <RS256 JWT>` header (audience = app name, issuer `rep-delivery`, 1hr expiry), per-message-type mutation (`addAttempt`, `addSkillChange`, `addContextChange`, `addContentChange`, `addPositionChange`; `CtContentPositionMessage`/navigation is documented by CLAUDE.md as "currently skipped" but `AddPositionChange` mutation and `processPositionChange` function both exist and are wired in `routeDataToApplications` — (README-stated "skipped" contradicts code, which does process it; named finding, direction: doc may be stale or code changed since doc was written, cannot tell which from this probe alone). This entire seam is lumilo-bridge-specific dispatch logic that the ripple is replacing for lumilo-bridge only — lid-backend keeps this seam.
- **rep-delivery → Secrets Manager (signing key)**: `getSigningKey()` in `services/secretsService.js` — seam exists but this probe did not read secretsService.js; named as an unconfirmed gap, not asserted.
- **deploy.sh → CloudFormation**: passes `HttpDeliveryEndpoints`/`CTP10_STREAM_NAME`/`OUTPUT_STREAM_NAME`/`SIMULATOR_STREAM_NAME` as stack parameters, but `infrastructure.yaml`'s actual `Parameters:` block defines `RepDeliveryEndpoints` (not `HttpDeliveryEndpoints`) and no stream-name override parameters at all (`Ctp10StreamName`, `OutputStreamName`, `SimulatorStreamName` exist but nothing in deploy.sh's `extract_config_values` sets `$CTP10_STREAM_NAME`/`$OUTPUT_STREAM_NAME`/`$SIMULATOR_STREAM_NAME` — those bash variables are referenced in `deploy_stack()` but never assigned anywhere in the script). Confirmed by reading deploy.sh in full: this local-testing/reference deploy path looks broken or superseded, reinforcing telos.md §2's "don't treat infrastructure.yaml/deploy.sh as source of truth."

## §5 Named findings
- **(code-derived)** `rep-delivery`'s tenant/class matching is OR, not AND, contradicting this repo's own CLAUDE.md. `lives-in: rep-delivery/config/index.js`
- **(code-derived)** `infrastructure.yaml`'s inline `RepDeliveryLambda` (Python, generic HTTP webhook forwarder) does not match the real `rep-delivery/index.js` (Node.js, GraphQL, JWT). The CloudFormation template in this repo is not what's actually deployed for that Lambda. `lives-in: infrastructure.yaml, rep-delivery/index.js`
- **(code-derived)** `infrastructure.yaml` pins `RuntimeEnvironment: FLINK-1_15` for the Kinesis Data Analytics application, while `rep-flink/pom.xml` builds against Flink 1.20.0 — a version mismatch between the declared runtime and the actual build target. `lives-in: infrastructure.yaml, rep-flink/pom.xml`
- **(code-derived)** `config/schema.json` requires a `streams` object that no environment config file (`dev.json`, `prod.json` — both read in full) actually has. Schema and config have diverged; schema cannot currently validate any real environment file. `lives-in: config/schema.json, config/environments/`
- **(code-derived)** `deploy.sh` references CloudFormation parameter name `HttpDeliveryEndpoints` and shell variables `$CTP10_STREAM_NAME`/`$OUTPUT_STREAM_NAME`/`$SIMULATOR_STREAM_NAME` that are never assigned in the script and don't match `infrastructure.yaml`'s actual parameter names (`RepDeliveryEndpoints`). `lives-in: deploy.sh, infrastructure.yaml`
- **(README-stated vs code, direction unclear)** CLAUDE.md says `CtContentPositionMessage` navigation events are "currently skipped," but `rep-delivery/index.js` has a fully wired `processPositionChange`/`ADD_POSITION_CHANGE_MUTATION` path in `routeDataToApplications`. `lives-in: rep-delivery/index.js`
- **(code-derived)** The real prod deployment mechanism is Jenkins (`.jenkins/build.Jenkinsfile`, `.jenkins/deploy.Jenkinsfile`) driving CloudFormation stacks defined in an external, not-checked-out-here infrastructure repo (`baseDir: "lambdas/realtime-event-provider"`), not this repo's `infrastructure.yaml`/`deploy.sh`. Those two files are best treated as local-dev/reference artifacts only. `lives-in: .jenkins/deploy.Jenkinsfile`
- **(code-derived)** `@aws-sdk/client-secrets-manager` and `@aws-sdk/client-kinesis` are pinned to different minor versions across `rep-delivery` (^3.450.0) vs `rep-key-rotation`/`rep-simulator` (^3.864.0) — no functional issue observed, but a naive "keep dependencies in sync" port instinct would be wrong here; the skew appears to be organic (never revisited), not intentional pinning. `lives-in: rep-delivery/package.json, rep-key-rotation/package.json, rep-simulator/package.json`
