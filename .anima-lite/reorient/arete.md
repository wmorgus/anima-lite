## Part 1 — the resolution (original record, 2026-07-07, PIN-33)

Canonical operative text is `RESOLUTION.md` + `.claude/skills/ari-intake/SKILL.md` GATE-TELOS apex layer; this section is the design record.

### What it names

Every repo the custodian works on has, implicitly, one final claim above all others: a one-sentence description of what the repo *is for*, sovereign over its own telos doc. anima-lite's telos.md and the four-cause spine describe the repo in detail; the resolution names it in one breath, at the top of the authority stack. It caps the identity work in `identity.md` and the four-cause frame in `four-causes.md`: the final cause has a final cause of its own.

**Authority inversion.** The sentence is ground truth. The code is fallible embodiment of it. `telos.md` is fallible commentary on it — stale-able, hash-checked, correctable. A mismatch between code and the resolution indicts the code or the workstream, never the sentence, by default. "Live" means continuously-checked accuracy, not frequent edits — the resolution is maximally ember: it changes only by constitutional growth, and every intake re-verifies its accuracy rather than assuming it.

### Ratified decisions (2026-07-07, operator round)

1. **Home.** The resolution lives in its own artifact at repo root, `RESOLUTION.md` — the sentence plus a ratification line (date + operator), nothing else. Smallest file, highest authority; other docs cite it, never restate it. anima-lite only for now — target repos getting one is future work, not built.
2. **Text.** anima-lite's resolution sentence: "agentic harness using aristotelean decomposition to build software that admits change only as argument." States an enforced property (nothing enters the pipeline unargued), not an aspiration.
3. **Mechanism.** The per-intake terminal check — does the repo this change produces still answer to the resolution? — folds into GATE-TELOS as its apex layer: ari-intake reads `RESOLUTION.md` first, then `telos.md`. One gate, two authority layers; no new gate ID. Bootstrap posture: if the target repo has no `RESOLUTION.md`, the apex layer is recorded absent, never fabricated.
4. **Adjudication (constitutional).** A resolution-layer failure is named **drift** (the change/code is wrong — fix it, ordinary custodial work) or **growth** (the resolution itself must change). Growth is never bundled: a resolution edit is its own workstream — separate intake, separate commit, explicit operator ratification recorded in `RESOLUTION.md`'s ratification line — and the change that provoked the divergence stays blocked until the resolution question resolves.

### The guard

Founder-exemption trap shape: the risk isn't that the resolution is wrong, it's that the sentence gets quietly rewritten to match whatever the code has already drifted into — laundering drift as growth so no one has to fix anything. The adjudication fork exists specifically to block that move: growth requires its own workstream and explicit operator ratification, on the record, every time. A silent `RESOLUTION.md` edit is never a valid outcome of ordinary work.

### Capture-only: generalization (superseded by Part 2)

The resolution concept generalizes beyond anima-lite — every repo could carry a one-sentence sovereign description above its own telos doc. This was recorded 2026-07-07 as ratified direction, not built: *"no machinery for other repos exists yet; no target repo has one."* Part 2 below is that machinery, arrived at from the other direction — not "let's build the missing piece" but "lumilo-bridge is context-starved, it needs a founding telos" — landing on the same concept independently, then recognizing the connection.

## Part 2 — arete, the work-type (2026-07-14)

Prompted by lumilo-bridge's context starvation: it has material/efficient cause (moves data) but no final cause on record — no one ever wrote down what it's *for*. Named concept: **arete** — founding a telos as the excellence-standard a tool is meant to serve, then judging existing code against it, rather than reading the telos off whatever the code currently does.

Relationship to what exists: closer to ripple's posture toward proto ("validates the contract, does not become it") than to how `ari-map` normally reads a repo (code as ground truth for the spine). Ripple institutes a *new promise* across N repos from an authored contract; arete institutes a *founding telos* in a repo (or repos) that never had one, and judges the existing implementation against it. Different starting condition — ripple's repos are usually promise-shaped already (a feature to add); arete's target has no defending telos at all, which is why "context starvation" is the right diagnosis, not drift.

### Rulings (2026-07-14)

1. **Fifth work-type, with dedicated skills — not a posture-variant of the existing five.** Port/ripple/debt-work/harness-change all run through the same skills (intake/map/argue-rhetoric/code-rhetoric, plus diagnose for debt-work) with a different posture at each stage. Arete gets its own skills, purpose-built for its shape. Deserves its own flowchart — folded into PIN-42 (each work-type, arete included, gets a dedicated flowchart; the current unified one is too dense to hold a fifth).
2. **The founding artifact is the arete statement — the resolution concept (Part 1) generalized.** Not a third artifact type sitting beside `intent.md` and `RESOLUTION.md` — this *is* Part 1's missing machinery, now with a real mechanic for producing it, and a name that carries more than "resolution" did (excellence-as-standard, not just settled sentence). Sibling to `intent.md` in the sense that it's a first-class artifact, but its epistemic status is different: `intent.md` is argued-for (defended, by prototype or language); the arete statement is asserted-as-standard (a ratified value judgment, no defense required beyond the operator's own conviction).
3. **Telos authoring is iterative and cascades top-down.** Mechanic (operator-specified):
   - **Pre-pass, operator obligation: seed the context.** The single most important epistemic-janitoring step — better seed, better outcome, everything downstream falls out of it. The seed is the living project's origin point, what everything else grows around.
   - **Seed-context gate, hard precondition, not advisory.** The system enforces starting best practices before an arete pass is allowed to run at all — e.g., an index file labeling what everything is. Fails the check → refuses to run, full stop. Starting rule list is small and will grow as gaps are found (ratified 2026-07-14) — not written exhaustively up front.
   - **The pass itself:** agent reads the seed context in full, asks clarifying questions, drafts a v1 arete statement. The statement is **caveman-dense** — concise, conceptually maximal, not prose. Operator and agent iterate by language until it matches intent; operator ratifies; it's committed.
   - **Cascades top-down.** Run first at the system level (e.g. the whole Lumilo project, seeded by its grant proposal + context) to fix the parent arete statement. Then re-run per component, downstream, deriving each child's arete statement from the parent — repo by repo (realtime-event-provider, lumilo-bridge, the XR client, etc.). Codebases are expressions of upstream teloses, not independently founded from scratch.
   - This reshapes scope (ruling 6): "single repo first" means proving the cascade on the smallest real slice — one parent statement, one derived child — not a component founded with no parent at all.
4. **New judgment, adjacent to `ari-diagnose` but not a mode of it.** `ari-diagnose`'s four primitives (spec/epistemic/world-drift/craft) all presuppose an existing telos the code drifted *from*. Arete's code never had one to drift from — founding-into-nothing is a different frame, the same category distinction that kept ripple from being "greenfield + port."
5. **Own gate before execution, preceded by iteration.** Everything arete's judgment is cutting, ratified as a batch, before any code moves — separate from argue-rhetoric's normal per-claim confirm. These claims die by the *absence* of a defending telos, not by an argued change to them; that's a different kind of judgment and deserves to be seen as a batch, not buried in per-claim confirms. The batch gate is not the operator's first exposure to the cut, though: `ari-arete-pan` produces a draft cut, presents it in full, and lets the operator iterate — reassign an item between nugget/ore/slag, sharpen an ore pressure-note — before the gate fires on the stabilized result. Mirrors `ari-arete`'s own draft→iterate→ratify pattern for the statement itself (ruling 3), rather than the cut arriving cold at a single yes/no. Amended 2026-07-14, during the arete-skills contracting session, on the operator's own observation.
6. **Single repo first, reshaped by ruling 3's cascade.** Prove the mechanic on the smallest real slice: one system-level arete statement (Lumilo), one component-level statement derived from it (lumilo-bridge) — not lumilo-bridge founded in isolation with no parent.

7. **Cut-judgment verdict, three-way, using anima's forging vocabulary — not meat/scar-tissue/dead-weight literally.**
   - **Nugget** — cleanly serves the arete statement as authored. Kept; still stated and confirmed via argue-rhetoric so "kept as-is" is an examined decision, not a silent survivor.
   - **Ore** — a real pressure produced this, but the current form doesn't serve the telos cleanly. The judgment records *what pressure produced it* as a note (the fodder — what stops the rewrite from reinventing a solution to an unnamed pressure), then the claim is rewritten against the arete statement. Same word as shitcorping's "ore" (`corpora/anima/process.md`, unrefined claim awaiting smelting) — same shape (raw, not yet valuable as-is, needs transformation, not discarded), different pipeline. Left as an intentional echo, not a collision, unless it proves confusing in practice. Anticipated common case, noted 2026-07-14 (operator observation): throwaway utility scripts — written to get one specific thing done because nothing easier existed at the time — are canonically ore. No defending telos, but a real, nameable pressure behind them; `ari-arete-pan` should expect to find plenty of these, not treat them as an edge case.
   - **Slag** — doesn't serve the telos, no signal worth carrying forward. Discarded outright.
8. **Execution target is a new, clean repo — not building in place.** Same geometry ripple already uses toward proto, direction inverted: ripple's proto is *trusted* code that validates a contract without becoming it; arete's existing repo is *untrusted* code that gets judged (nugget/ore/slag) without becoming the new repo either. The old repo stays untouched and read-only — `ari-map`'s spine of it is the judgment's evidence, and after arete it's historical/reference material, not a live target. The new repo is built claim-by-claim: nuggets get their claim stated and confirmed (even though the code doesn't change), ore gets rewritten from its claim, slag never crosses over. Reuses existing machinery directly — the new repo is just a code-rhetoric leg starting from empty instead of from existing code to translate. Building in place was rejected: it tempts exactly the destructive-edit-to-shared-history failure mode arete exists to avoid.
9. **Cutover is out of scope for arete, permanently — not deferred, excluded.** Deploy config, DNS/service pointers, downstream consumers of the old repo, secrets, CI migration: real problems, but operational/infra decisions, not claim-authorship ones. Arete's job ends when the new repo's claims are built and validated; getting traffic/consumers pointed at it is a separate workstream every time.

10. **Skill set, named:** `ari-arete` (bare — asserts, doesn't argue; seed-gate, iterative draft-and-ratify pass, cascade re-invocation per node) and `ari-arete-pan` (bare; sorts existing code into nugget/ore/slag against an arete statement, holds the batch gate). `ari-argue-rhetoric`/`ari-code-rhetoric` gain an arete posture (new-empty-repo target, ruling 8) rather than new skills — same pattern ripple and harness-change used.
11. **`ari-arete-pan` is an independent entry point, not just the next step after `ari-arete` in one session.** Its precondition is "an arete statement exists for this repo," not "an `ari-arete` pass just ran" — it can be invoked standalone later: re-panning a repo against an unchanged statement (periodic drift check, arete's version of `ari-map`'s re-probe), or panning against a statement authored in an earlier or different session entirely. This is itself the argument for keeping it a separate skill rather than a second mode of `ari-arete` — a mode implies it's reached through the parent skill's flow; an independent entry point has its own precondition and its own reason to be called directly.
